ZKNOT™ Docs v1.0

Technical reference for the ZKNOT™ attestation API, attestation artifact schema, ZK-LocalChain™ chain format, and product integration specifications. All records produced by ZKNOT™ hardware are independently verifiable without contacting ZKNOT™ infrastructure.

ℹ
The verification API at api.zknot.io and the SDK are under active development. This document reflects the v1 specification. Contact ops@zknot.io for early API access.

Core Concepts

Short Code

A human-readable identifier assigned to every chain entry at write time. Format: ZK-XXXX-XXX — four alphanumeric characters, a hyphen, three alphanumeric characters. Printed on TrustSeal™ labels and included in PDF reports. Resolves to the full attestation artifact via the verify API or at verifyknot.io.

Attestation Artifact

The complete JSON record produced at each chain write event. Contains the short code, artifact type, device ID, timestamp, chain position, artifact hash, entry hash, and device signature. The artifact is the unit of evidence — everything else derives from it.

Chain Entry

An attestation artifact written to ZK-LocalChain™. Each entry includes a prev_hash linking it to the prior entry, making every entry a mathematical proof of all prior entries. Entries are append-only — no delete, no edit operation exists.

Device Signature

Every chain entry is signed by a ZKKey™ hardware device using ECDSA P-256. The private key is hardware-locked inside the ATECC608B secure element and cannot be exported. Verification requires only the public key, which is public by design.

→
Independent verification requires only: the attestation artifact JSON, the device public key, and a SHA-256 library. ZKNOT™ infrastructure is not required to verify any record.

Error Codes

SHORT_CODE_NOT_FOUND
404
The short code does not exist in the chain. It has never been written, or was entered incorrectly.
CHAIN_INTEGRITY_FAILURE
409
The entry's hash does not match its position in the chain. This indicates tampering. This error should never occur in normal operation.
SIGNATURE_INVALID
422
The device signature on this entry does not verify against the registered device public key.
DEVICE_NOT_FOUND
404
The device ID is not registered. All deployed ZKKey™ devices must be registered before use.
RATE_LIMITED
429
Verification API is rate-limited at 60 requests/minute per IP for unauthenticated callers.

Attestation Artifact Schema

The canonical JSON schema for all ZKNOT™ chain entries. Version 1.0. Every field is required unless marked optional.

schema_version
string
Schema version. Currently "1.0". Increment on breaking changes.
short_code
string
Human-readable identifier. Format: ZK-[A-Z0-9]{4}-[A-Z0-9]{3}. Globally unique per chain entry.
artifact_type
enum
COMBINED_SESSION · TRUSTSEAL_EVENT · ZKKEY_SIGNATURE · POWERVERIFY_CONNECT · CHAIN_EXPORT
device_id
string
The ZKKey™ device that produced this entry. Resolves to a public key via GET /v1/device/{device_id}.
signed_at
string (ISO 8601)
UTC timestamp of the signature event. Set by the ZKKey™ hardware clock at button-press time.
chain_position
integer
Monotonically increasing position in the chain. Starts at 1. Gaps indicate a chain integrity failure.
chain_length_at_signing
integer
Total chain length at the moment of signing. Must equal chain_position for a valid head entry.
prev_hash
string (hex)
SHA-256 hash of the immediately preceding chain entry's full JSON. null for genesis entry only.
artifact_hash
string (hex)
SHA-256 hash of the digital artifact associated with this entry (image, file, session). The proof that the artifact is unchanged.
entry_hash
string (hex)
SHA-256 hash of this entry's full JSON (excluding the entry_hash field itself). Used as prev_hash in the next entry.
device_signature
object
ZKKey™ hardware signature over this entry. See Device Signature schema below.
verification_url
string (URI)
Canonical public verification URL for this entry. Always https://verifyknot.io/{short_code}.
metadata
object (optional)
Optional key-value extension for operator notes, GPS coordinates, seal ID, case number. Not included in signature computation.

Device Signature Schema

algorithm
string
Always "ECDSA-P256" in v1. Per NIST FIPS 186-4.
public_key_fingerprint
string
Colon-separated hex fingerprint of the device public key. For quick visual identification.
public_key_pem
string
Full PEM-encoded public key. Included for offline verification — no network call required.
signature
string (base64)
DER-encoded ECDSA signature, base64url-encoded. Verifies against the entry_hash field.

Independent Signature Verification

Python
from cryptography.hazmat.primitives import hashes, serialization
from cryptography.hazmat.primitives.asymmetric import ec
import base64, hashlib, json

# Load the attestation artifact
artifact = json.load(open("artifact.json"))

# 1. Verify the entry hash
entry_copy = {k: v for k, v in artifact.items() if k != "entry_hash"}
computed = hashlib.sha256(
    json.dumps(entry_copy, separators=(",", ":"), sort_keys=True).encode()
).hexdigest()
assert computed == artifact["entry_hash"], "Entry hash mismatch"

# 2. Verify the device signature
sig = artifact["device_signature"]
pub_key = serialization.load_pem_public_key(
    sig["public_key_pem"].encode()
)
pub_key.verify(
    base64.urlsafe_b64decode(sig["signature"]),
    bytes.fromhex(artifact["entry_hash"]),
    ec.ECDSA(hashes.SHA256())
)

# 3. Verify chain linkage (if you have prev entry)
prev_hash = hashlib.sha256(
    json.dumps(prev_artifact, separators=(",", ":"), sort_keys=True).encode()
).hexdigest()
assert prev_hash == artifact["prev_hash"], "Chain break detected"

print("✓ Artifact verified. Chain intact.")

Chain Format

Structure

ZK-LocalChain™ is a linear, append-only sequence of attestation artifacts. Each entry references the hash of the prior entry, forming a cryptographic chain. Any modification to any entry invalidates every subsequent entry's prev_hash — making tampering detectable by any party with two consecutive entries.

Hash Computation

The entry_hash for each entry is computed as:

Algorithm
// 1. Take the full entry JSON
// 2. Remove the "entry_hash" key
// 3. Canonicalize: sort keys, remove whitespace
// 4. UTF-8 encode
// 5. SHA-256 hash → hex string

entry_hash = SHA256(
    JSON.stringify(entry_without_entry_hash, {
        keys: sorted,
        separators: (",", ":")
    })
).hex()
⚠
Key ordering is significant. Always use sort_keys=True (Python) or equivalent canonical JSON serialization. Whitespace differences will produce a different hash.

Independent Chain Verification

To verify an entire exported chain without ZKNOT™ infrastructure:

Python
import json, hashlib

entries = json.load(open("chain_export.json"))  # list, position order

for i, entry in enumerate(entries):
    # Verify entry hash
    e = {k: v for k, v in entry.items() if k != "entry_hash"}
    computed = hashlib.sha256(
        json.dumps(e, separators=(",",":"), sort_keys=True).encode()
    ).hexdigest()
    assert computed == entry["entry_hash"]

    # Verify chain linkage
    if i > 0:
        assert entry["prev_hash"] == entries[i-1]["entry_hash"]

    # Verify chain position
    assert entry["chain_position"] == i + 1

print(f"✓ Chain verified. {len(entries)} entries. Integrity intact.")

ZKKey™ — Technical Reference

Hardware cryptographic signing device. Human-gated. Post-nonce signing structurally impossible.

Secure Element
hardware
Microchip ATECC608B · ECC-P256 · ECDH · SHA-256
Host MCU
hardware
STM32F411 · ARM Cortex-M4 · 100MHz
I²C Address
0x60
Default ATECC608B I²C address on PB6/PB7 (I2C1)
Signing Algorithm
ECDSA-P256
Per NIST FIPS 186-4. Key slot 0. Non-exportable.
Interface
USB-A / I²C
USB HID + Serial. 4-wire I²C header for embedded integration.
Nonce enforcement
hardware timer
Tamper-evident. Non-resettable without new chain entry.
Physical trigger
button press
PC13 GPIO. No remote signing path exists in firmware.

PowerVerify™ — Technical Reference

Physical write-path elimination device. No firmware. Continuity-testable.

Write protection
physical
Write data lines severed at PCB trace level. No firmware involved.
Testability
multimeter
Open circuit on write lines. Measurable with standard continuity test in any lab or court.
Interfaces
USB 3.0 / SATA
M.2 variant in development. Passive passthrough form factor.
Firmware
none
Zero firmware. No attack surface below the physical protection layer.
Failure mode
fail-safe
Hardware failure results in loss of read path — never restoration of write path.

ZK-LocalChain™ — Technical Reference

Offline-capable append-only hash-chained ledger. No central authority required.

Storage
W25Q flash
SPI NOR flash. Encrypted at rest. Field-replaceable.
Hash algorithm
SHA-256
Per entry and for chain linkage. Hex-encoded in JSON.
Connectivity
offline-capable
Full operation with zero network. Syncs to verifyknot.io on reconnection.
Append policy
append-only
No delete or edit operation exists at any layer of the stack.
Export format
JSON / PDF
Attestation artifact JSON array or signed PDF chain-of-custody report.

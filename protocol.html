Walk the protocol
Step 01 · Hardware

Connect PowerVerify™

Before any data is touched, PowerVerify™ is inserted between the evidence device and the forensic workstation. The write data path is physically severed at the PCB level. From this point forward, the evidence device cannot be written to — not by the operator, not by the OS, not by any software on the workstation. This is not a policy. It is a physical fact testable with a multimeter.

→
Write path eliminated before any software runs
→
No firmware to exploit or misconfigure
→
Continuity-testable in any court or lab

Step 02 · Acquisition

Forensic Capture

With the write path eliminated, data acquisition proceeds using standard forensic imaging tools. Because PowerVerify™ operates below the software layer, any imaging tool — dd, FTK, Cellebrite, custom — can be used without modification. The tool cannot accidentally write to the source because the hardware does not permit it. The resulting image or file set is hashed before proceeding.

→
Compatible with any forensic imaging tool
→
SHA-256 hash computed on captured artifact
→
Hash becomes the binding element for all subsequent steps

Step 03 · Physical Binding

Apply TrustSeal™

A TrustSeal™ is applied to the evidence device packaging. At the exact moment of application, the seal's unique ID is scanned and a chain entry is written that binds the seal ID, the artifact hash from Step 02, the operator device ID, and the timestamp into a single immutable record. Physical custody and digital custody merge here and cannot be separated afterward.

→
Seal application and chain entry are atomic — one event
→
Tamper reveals permanent VOID pattern on surface and backing
→
Short code on seal face resolves at verifyknot.io

Step 04 · Hardware Signing

ZKKey™ Sign

The operator presses the physical button on ZKKey™. This is the only action that triggers a signature. No remote command, no scripted automation, no administrator override can produce a valid signature. The ATECC608B secure element signs the complete chain entry payload using ECDSA P-256. The private key never leaves the hardware. After the nonce expires, no further signatures are possible for this session.

→
Physical button press is the only valid trigger
→
Private key is hardware-locked — non-exportable
→
Post-nonce signing structurally impossible

Step 05 · Ledger

Chain Record Written

The signed entry is written to ZK-LocalChain™. Its hash is computed and chained to the previous entry — making every prior record part of the mathematical proof of this one. The chain operates offline-capable; no network connection is required. When connectivity is available, the entry is published to verifyknot.io for public resolution. Any party can independently verify the entire chain from any exported entry.

→
Entry is append-only — no delete, no edit possible
→
Hash-chained to every prior entry in the ledger
→
Independently verifiable — ZKNOT™ not required to verify

Protocol Guarantees

What the protocol guarantees — and how.

Not claims. Not features. Structural facts — each provable by a different method.

01
The source device was not modified
Provable by PowerVerify™ continuity test — a multimeter confirms the write path was severed before the workstation was connected. No software claim required.
02
The captured artifact is unchanged
The SHA-256 hash in the chain entry can be recomputed against the artifact at any time. If the hash matches, the artifact is byte-identical to what was captured.
03
The physical item was sealed at this moment
The TrustSeal™ chain entry binds the seal ID, artifact hash, operator, and timestamp. Seal removal reveals permanent void evidence. The chain entry remains either way.
04
A human was physically present for each signature
ZKKey™ requires a physical button press for every signature. No remote command, script, or administrator credential can substitute. The hardware enforces this.
05
The chain has not been modified since recording
Any retroactive modification to any chain entry breaks every subsequent hash. The math makes silent tampering computationally impossible — detectable by any party with the chain.
06
Any party can verify — without ZKNOT™
Short codes on TrustSeal™ labels and PDF reports resolve at verifyknot.io. No account. No vendor call. No trust in ZKNOT™ required to confirm every guarantee above.
